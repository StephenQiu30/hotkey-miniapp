# AI 热点监控工具验收标准

本验收标准对应从零重建后的轻量 MVP，不验收旧实现兼容性、旧数据库迁移或旧 OpenAPI 契约兼容性。
新增企业级 P0 任务时，以本文件为执行闭环入口，并与 `docs/product/prd/00-企业级AI热点监控平台PRD.md` 对齐。

更细的剩余验收分类见 [MVP 验收差距清单](../验收差距清单.md)。

## 企业级 P0 追加验收（2026）

### A. 可扩展核心架构

- [ ] 存在 Provider 注册机制（`BaseProvider` + 工厂/注册表）并有至少 2 个可插拔来源实现接入主流程。
- [ ] 存在 LLM Provider 抽象（`BaseLLMProvider`）并可切换模型供应商（至少 1 条可运行路径）。
- [ ] 核心抓取与 AI 分析链路支持异步执行（后台任务/协程）与失败隔离。
- [ ] Redis 缓存与任务/热点命中率、热点级别命中率可观测。

### B. 安全与权限

- [ ] JWT 登录/鉴权链路可验证访问受限接口。
- [ ] RBAC 最小权限用例可执行（管理员与普通用户至少两类角色）。
- [ ] API 限流规则可配置且可观测限流触发。

### C. 内容交付

- [ ] RSS 端点可输出关键词 RSS 与全局热门 RSS（至少 `keyword/user/trending` 3 类以上）。
- [ ] 日报/周报可输出 Markdown；必要时支持 HTML 预览路径。
- [ ] 热点明细页含 AI 分析字段版本化与去重依据（来源、标题归一化、URL）。

### D. 可视化与运营

- [ ] 热点趋势图、热度排行、来源占比、情绪分布任意 4 个图表组件可用。
- [ ] 系统健康/任务成功率/失败率与关键错误日志可查询。
- [ ] 运营控制台具备用户、关键词、来源、报告和任务的管理入口。

## 1. 已通过自动化或静态确认

- [x] FastAPI 后端入口位于 `apps/api`。
- [x] Next.js 前端位于 `apps/web`。
- [x] PostgreSQL 是唯一 P0 数据库。
- [x] `sql/001_init_schema.sql` 是 schema 事实源，包含 keywords、sources、hotspots、ai_analyses、notifications、check_runs、reports、settings。
- [x] SQLAlchemy models 与 `sql/001_init_schema.sql` 使用同一组核心表和字段。
- [x] RSS、Hacker News、X/Twitter、Bing、Bilibili、Sogou-style 具备 adapter 代码路径。
- [x] X/Twitter 未配置 Token 时记录跳过错误，不中断整体流程。
- [x] Bing 未配置 Key 时记录跳过错误，不中断整体流程。
- [x] AI 查询扩展可在本地降级路径生成 2-5 个去重查询词。
- [x] AI 分析结果包含真实性、相关性分数、相关性理由、关键词是否命中、重要性和摘要。
- [x] `active` / `filtered` 阈值过滤规则已实现：`relevance_score >= RELEVANCE_THRESHOLD` 且 `is_real is not False` 才进入 `active`。
- [x] `filtered` 热点不发送事件邮件、不进入日报/周报。
- [x] SMTP 缺失时系统仍可运行，并记录 `skipped`。
- [x] `/api/search` 可执行即时搜索，搜索结果不入库。
- [x] `/api/reports` 可生成 daily/weekly 报告，并只聚合 `active` 热点。
- [x] `/api/daily-reports` 不再注册，不保留兼容别名。
- [x] 前端提供工作台页面：热点、搜索、关键词、来源、任务、报告、通知和设置。
- [x] `/`、`/pricing`、`/app`、`/app/search`、`/app/reports` 已完成桌面与移动浏览器布局走查。

## 2. 需真实配置后验收

- [ ] 空 PostgreSQL 通过 `sql/001_init_schema.sql` 初始化并完成一次端到端检查。
- [ ] 使用真实 X API v2 Recent Search Token 验证 X/Twitter 来源。
- [ ] 使用真实 Bing Search Key 验证 Bing 来源。
- [ ] 使用真实 OpenAI 兼容模型配置验证查询扩展和热点分析。
- [ ] 使用真实 SMTP 配置验证事件邮件和报告邮件发送。
- [ ] 后端和 PostgreSQL 同时启动后，使用浏览器验证工作台真实数据创建、搜索、报告生成和发送链路。

## 3. 人工功能验收

- [ ] 可以创建、更新、删除、启停关键词。
- [ ] 同一 URL + 来源组合不能重复入库。
- [ ] 热点可按关键词、来源、重要性、时间范围筛选。
- [ ] 热点可按时间、相关性或重要性排序。
- [ ] 可手动触发热点检查。
- [ ] 可简单定时触发热点检查。
- [ ] 报告内容以本地 Markdown 模板生成；AI 报告增强不作为 P0 必需能力。

## 4. 非目标确认

- [x] P0 不引入多租户。
- [x] P0 不引入复杂权限。
- [x] P0 不引入计费。
- [x] P0 不引入实时推送。
- [x] P0 不引入向量库。
- [x] P0 不强制依赖 Celery/Redis。
- [x] P0 不兼容旧表结构、旧内存仓库或旧 bootstrap 示例数据。
