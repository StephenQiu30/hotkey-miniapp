# 监控告警闭环 PRD（B3）

## 1. 背景

项目已有 `/api/ops/metrics` 快照能力，但没有稳定落地到阈值告警与处置流程。企业级 P0 在运行阶段最少需要可验证的告警闭环。

## 2. 目标

- 固化告警指标基线与采样口径；
- 输出首版告警阈值策略并形成发布版本；
- 完成至少一次可控告警演练，形成触发、确认、处理、恢复闭环记录。

## 3. 非目标

- 不引入第三方告警平台；
- 不实现复杂多渠道（短信/IM）矩阵；
- 不做基于机器学习的自适应阈值。

## 4. 功能定义

- 指标盘点
  - 抓取成功/失败、任务失败率、429 次数、SMTP 发送成功率、AI 调用错误率。

- 告警策略
  - 输出可版本化阈值建议（初版保守值）。

- 演练闭环
  - 通过人为触发/模拟故障验证告警通知、应答时间和恢复动作。

## 5. 风险与边界

- 样本不足导致阈值误报偏高；
- 通知渠道依赖外部邮件服务波动；
- 告警误判时需保留人工确认流程，不得自动停服。

## 6. 交付与验收

- 提交 `docs/plans/19-监控告警最小闭环演练计划.md`。
- 提交 `docs/plans/21-告警阈值与变更流程计划.md`，形成后续持续更新机制。
- 在验收记录中保留至少一次演练时间线与处理结果。
